Offline Operations Guide

가재코드 폐쇄망 사용 가이드

WSL2와 tmux에서 가재코드를 실행하고, 작업 성격에 맞는 Skill을 선택하며, 중단된 작업을 복구하는 방법을 정리한 운영 문서입니다.

실행 방법 작업 흐름 Skill과 규칙 tmux 사용법 검증과 복구

1. 처음 실행하기

설치 완료 후 바탕 화면의 GajaeCode diagnostics.cmd를 먼저 실행해 설치 상태를 확인합니다. 문제가 없으면 GajaeCode tmux.cmd를 실행합니다.

기본 실행: 바탕 화면 바로가기는 gjc를 gjc라는 이름의 tmux 세션에서 실행합니다. 창을 닫더라도 tmux 세션이 유지되면 같은 작업 화면으로 다시 접속할 수 있습니다.

특정 프로젝트 폴더에서 시작

프로젝트 경로를 직접 선택하려면 WSL 터미널에서 다음 순서로 실행합니다. Windows의 C:\work\sample은 WSL에서 /mnt/c/work/sample로 접근합니다.

wsl -d GajaeCode
source ~/.config/gajae-code/env
cd /mnt/c/work/sample
tmux new -A -s sample
gjc

프로젝트마다 tmux 세션 이름을 다르게 사용하면 여러 작업 화면을 구분하기 쉽습니다. 단, 현재 설치의 에이전트 병렬도는 단일 내부 모델 서버의 안정성을 고려해 제한되어 있습니다.

2. 권장 작업 흐름

명확하고 작은 작업

Skill 없이 바로 작업을 지시합니다. 조사, 수정, 테스트, 결과 보고까지 한 요청에 명시합니다.

요구사항이 모호한 작업

/skill:deep-interview로 질문과 합의 과정을 먼저 진행합니다.

범위가 크고 계획이 필요한 작업

/skill:ralplan으로 구현 계획과 위험을 검토한 뒤 계획을 승인합니다.

완료 증거가 필요한 장기 작업

/skill:ultragoal로 목표 분해, 구현, 테스트, 검증 루프를 실행합니다.

작업 지시 예시

작은 버그 수정:

로그인 실패 원인을 코드와 로그에서 조사해줘.
원인을 확인한 뒤 영향 범위가 작은 수정안을 적용하고,
관련 테스트를 실행한 다음 변경 파일과 검증 결과를 보고해줘.

모호한 기능 요구사항 구체화:

/skill:deep-interview "관리자 권한 변경 기능의 요구사항, 예외 조건, 감사 기준을 구체화해줘"

구현 계획 검토:

/skill:ralplan "승인된 관리자 권한 요구사항을 구현 계획으로 분해하고 누락과 위험을 검토해줘"

계획 기반 구현과 검증:

/skill:ultragoal "승인된 계획을 목표로 분리하고 구현, 테스트, 최종 검증까지 완료해줘"

ultragoal 상태 명령

gjc ultragoal status
gjc ultragoal create-goals --brief-file <승인된-계획-파일>
gjc ultragoal complete-goals
Skill은 상황에 따라 명시적으로 호출하는 것이 가장 확실합니다. 모델이 작업 내용을 보고 도구나 하위 작업을 선택할 수는 있지만, 중요한 절차를 항상 보장하려면 작업 지시문에 사용할 Skill과 완료 조건을 직접 적으십시오.

3. 프로젝트 규칙 작성

프로젝트 루트의 AGENTS.md에는 모든 작업에 공통 적용할 규칙을 기록합니다. 작업별 요구사항과 승인된 계획은 별도 Markdown 문서로 관리하는 편이 안전합니다.

AGENTS.md 예시

# 공통 작업 규칙

- 변경 전에 관련 코드를 조사하고 기존 설계를 따른다.
- 작업 완료 전 빌드와 관련 테스트를 실행한다.
- 테스트 실패를 숨기거나 우회하지 않는다.
- API Key, 토큰, 내부 주소를 소스와 로그에 기록하지 않는다.
- 완료 보고에는 변경 파일, 테스트 명령, 테스트 결과를 포함한다.

AGENTS.md는 중요한 기준을 제공하지만 그 자체가 강제 실행 훅은 아닙니다. 반드시 지켜야 하는 검증은 Skill, 테스트 스크립트, CI, Git 훅 등 실행 가능한 하네스로 보강해야 합니다.

작업 문서 분리 예시

docs/tasks/admin-role-requirements.md
docs/plans/admin-role-implementation.md
docs/decisions/admin-role-audit-policy.md

4. 외부 Skill 등록

외부 Skill은 폴더 단위로 반입합니다. 최소 구성은 SKILL.md 하나지만, Skill이 사용하는 scripts, references, assets 폴더가 있다면 모두 함께 복사해야 합니다.

적용 범위WSL 내부 설치 위치
모든 프로젝트에서 사용~/.gjc/agent/skills/<skill-name>/SKILL.md
현재 프로젝트에서만 사용<project>/.gjc/skills/<skill-name>/SKILL.md
경로 주의: 현재 설치 버전의 전역 Skill 탐색 위치는 ~/.gjc/agent/skills입니다. 인터넷 문서의 이전 예시에 나오는 ~/.gjc/skills와 혼동하지 마십시오.

폐쇄망 반입 순서

  1. 외부 연결 PC에서 사용할 Skill 저장소를 특정 태그 또는 커밋으로 고정해 내려받습니다.
  2. SKILL.md와 포함된 스크립트를 검토해 외부 통신, 패키지 자동 설치, 토큰 수집, 임의 파일 삭제 명령이 없는지 확인합니다.
  3. 라이선스와 필요한 실행 프로그램을 확인합니다. 폐쇄망에서 추가 다운로드가 필요한 의존성은 미리 번들에 포함합니다.
  4. 검토한 Skill 폴더를 승인된 Git 저장소나 이동 매체로 폐쇄망 PC에 반입합니다.
  5. WSL의 전역 또는 프로젝트 Skill 폴더에 복사한 뒤 가재코드를 새로 시작합니다.

전역 디자인 Skill 설치 예시

Windows의 C:\GajaeCode-Skills\frontend-design에 Skill을 반입했다고 가정합니다.

wsl -d GajaeCode
source ~/.config/gajae-code/env
mkdir -p ~/.gjc/agent/skills
cp -R /mnt/c/GajaeCode-Skills/frontend-design ~/.gjc/agent/skills/
test -f ~/.gjc/agent/skills/frontend-design/SKILL.md
find ~/.gjc/agent/skills/frontend-design -maxdepth 2 -type f -print

설치 후 실행 중인 가재코드를 종료하고 다시 시작한 다음 명시적으로 호출합니다.

/skill:frontend-design "관리자 대시보드의 정보 구조와 시각 디자인을 개선해줘"

프로젝트 전용 설치 예시

cd /mnt/c/work/sample
mkdir -p .gjc/skills
cp -R /mnt/c/GajaeCode-Skills/frontend-design .gjc/skills/
test -f .gjc/skills/frontend-design/SKILL.md
gjc

SKILL.md 최소 형식

---
name: frontend-design
description: UI 화면, 디자인 시스템, 반응형 레이아웃을 설계하거나 개선할 때 사용한다.
---

# Frontend Design

## 작업 절차
1. 기존 화면과 프로젝트 디자인 규칙을 조사한다.
2. 정보 구조와 사용자 흐름을 먼저 정리한다.
3. 기존 프레임워크와 컴포넌트를 유지해 구현한다.
4. 데스크톱과 모바일 화면을 검증한다.
5. 변경 파일과 검증 결과를 보고한다.

name은 폴더명 및 /skill:<name> 호출명과 같게 유지하는 것이 좋습니다. description에는 이 Skill을 언제 선택해야 하는지 구체적으로 적습니다.

  • 모델이 작업에 맞춰 Skill을 발견하도록 하려면 명확한 description을 작성합니다.
  • 중요한 작업은 자동 선택에만 의존하지 말고 /skill:frontend-design처럼 명시적으로 호출합니다.
  • 전면의 YAML에 hide: true를 넣으면 자동 발견 목록에서는 숨고 명시적 호출로만 사용할 수 있습니다.
  • 같은 이름의 Skill이 여러 프로젝트 계층에 있으면 현재 작업 폴더에 가까운 프로젝트 Skill이 우선됩니다.

외부 Skill 검토 체크리스트

  • curl, wget, 원격 API, 텔레메트리 호출이 없는가
  • npm install, pip install 등 온라인 설치를 지시하지 않는가
  • API Key, 환경 변수, 사용자 파일을 출력하거나 전송하지 않는가
  • 광범위한 삭제, 권한 변경, Git 강제 푸시를 지시하지 않는가
  • 참조 파일과 스크립트가 Skill 폴더 안에 모두 포함됐는가
  • 고정된 버전과 원본 저장소, 라이선스를 기록했는가

5. 글로벌 AGENTS.md 등록

모든 가재코드 작업에 적용할 기본 규칙은 WSL 내부의 ~/.gjc/agent/AGENTS.md에 작성합니다. Windows의 문서 폴더가 아니라, 설치된 GajaeCode WSL 배포판의 gjc 사용자 홈 기준입니다.

파일 생성

wsl -d GajaeCode
mkdir -p ~/.gjc/agent
nano ~/.gjc/agent/AGENTS.md

nano 대신 Windows 탐색기에서 다음 경로를 열어 편집할 수도 있습니다.

\\wsl$\GajaeCode\home\gjc\.gjc\agent\AGENTS.md

글로벌 규칙 예시

# 모든 프로젝트 공통 규칙

- 응답과 작업 보고는 한국어로 작성한다.
- 변경 전에 관련 코드와 프로젝트 AGENTS.md를 먼저 확인한다.
- 기존 구조와 프레임워크를 우선 사용한다.
- 완료 전에 빌드와 관련 테스트를 실행한다.
- 테스트 실패나 미실행 항목을 숨기지 않는다.
- API Key, 토큰, 내부 서버 주소를 소스와 로그에 기록하지 않는다.
- 완료 보고에는 변경 파일, 실행한 명령, 검증 결과를 포함한다.
- 인터넷 연결이나 외부 패키지 설치가 필요하면 실행 전에 중단하고 보고한다.

규칙 적용 범위

파일용도
~/.gjc/agent/AGENTS.md모든 프로젝트 공통 기본 규칙
<project>/AGENTS.md해당 저장소와 하위 폴더의 프로젝트 규칙
<project>/.gjc/AGENTS.md가재코드에만 적용할 프로젝트 규칙
<subdirectory>/AGENTS.md특정 하위 모듈에 더 구체적인 규칙 적용

글로벌 파일에는 언어, 보안, 테스트, 보고 형식처럼 공통적인 규칙만 둡니다. 빌드 명령, 아키텍처, 폴더별 금지 사항은 프로젝트의 AGENTS.md에 기록하십시오. 규칙이 충돌하면 현재 작업 위치에 더 가까운 프로젝트 규칙을 구체적인 지침으로 취급하는 것이 안전합니다.

중요: AGENTS.md는 모델에 제공되는 지침이지 실행을 강제하는 보안 장치가 아닙니다. 반드시 실행돼야 하는 검사에는 테스트 스크립트, Git 훅, CI, ultragoal 검증 조건을 함께 사용하십시오.

파일을 추가하거나 수정한 뒤에는 기존 세션이 아닌 새 가재코드 세션을 시작해 규칙과 Skill 목록을 다시 읽게 하십시오.

6. tmux 필수 조작

아래의 Ctrl+b는 Ctrl 키와 b 키를 누른 뒤 손을 떼고, 다음 키를 누른다는 뜻입니다.

목적키 또는 명령
현재 세션에서 빠져나오기Ctrl+b 다음 d
세션 목록 보기tmux ls
기존 세션 재접속tmux attach -t gjc
없으면 만들고 있으면 접속tmux new -A -s gjc
새 창 만들기Ctrl+b 다음 c
다음/이전 창 이동Ctrl+b 다음 n / p
좌우 분할Ctrl+b 다음 %
상하 분할Ctrl+b 다음 "
분할 화면 이동Ctrl+b 다음 방향키
스크롤 모드Ctrl+b 다음 [, 종료는 q

tmux는 터미널 화면과 셸 프로세스를 유지합니다. 다만 모델 서버가 응답을 종료했거나 에이전트 프로그램 자체가 끝난 경우에는 tmux가 모델 작업을 자동 재개시키지는 않습니다. 이때는 같은 Skill이나 작업 지시를 다시 실행하고 저장된 상태를 확인해야 합니다.

7. 하위 에이전트와 병렬 실행

가재코드는 작업 성격에 따라 조사, 구현, 검증 역할을 하위 작업으로 분리할 수 있습니다. 현재 기본 설정은 내부 모델 서버의 안정성을 우선해 초기 동시 실행 수를 2, 재귀 깊이를 1로 제한합니다.

  • 두 하위 에이전트가 동시에 동작하더라도 H100 두 장에 각각 별도 모델이 올라간다는 뜻은 아닙니다.
  • 같은 Qwen 모델 엔드포인트를 여러 역할이 공유할 수 있습니다.
  • 병렬 실행의 효과는 모델 복제 수보다 서버의 동시 요청 처리량, KV 캐시, 배치 정책에 좌우됩니다.
  • Team 기능은 현재 기본 설치와 운영 범위에 포함하지 않습니다.

속도 저하, 응답 끊김, GPU 메모리 부족이 나타나면 병렬도를 올리지 말고 설치 진단 로그와 모델 서버 로그를 먼저 비교하십시오.

8. 검증, 중단 복구, 진단

기본 확인 명령

wsl -l -v
wsl -d GajaeCode -- bash -lc "gjc --version"
wsl -d GajaeCode -- bash -lc "tmux -V"
wsl -d GajaeCode -- bash -lc "gjc ultragoal status"

작업이 중간에 끊긴 경우

  1. 터미널 창만 닫혔다면 GajaeCode tmux.cmd로 다시 접속합니다.
  2. 가재코드가 종료됐다면 프로젝트 폴더에서 gjc를 다시 실행합니다.
  3. Skill 작업이었다면 같은 Skill을 다시 호출하고 기존 상태를 확인하도록 지시합니다.
  4. .gjc 아래 상태 파일을 직접 수정하기 전에 반드시 복사본을 남깁니다.

주요 상태 위치

위치용도
.gjc/specs요구사항과 명세
.gjc/plans작업 계획
.gjc/ultragoalultragoal 목표 및 진행 상태
.gjc/state실행 상태
%LOCALAPPDATA%\GajaeCode\diagnosticsWindows 설치 로그와 복구 자료

설치 보고서 판정

  • Passed: 해당 검사 정상 완료
  • Warning: 핵심 설치는 완료됐지만 운영 전 확인 필요
  • Failed: 로그와 RECOVERY.md를 확인한 뒤 설치 재실행 필요

9. 폐쇄망 운영 주의사항

  • API Key를 대화창, 소스 파일, Markdown 문서, Git 커밋에 입력하지 않습니다.
  • 업데이트 확인, 마켓플레이스, 별점 알림, 웹 검색, 브라우저 연동은 비활성화 상태를 유지합니다.
  • npm install, bun install, cargo install처럼 인터넷 의존성이 생길 수 있는 명령은 승인된 내부 저장소가 없으면 실행하지 않습니다.
  • Team 기능과 병렬도 증가는 내부 모델 서버의 동시 처리 검증 후 적용합니다.
  • 작업 완료 보고에는 실행한 테스트와 실패한 검사를 구분해서 기록합니다.
이 문서는 외부 글꼴, JavaScript, CDN을 사용하지 않는 폐쇄망용 단일 HTML 파일입니다.
